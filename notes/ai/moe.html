MoE — Mixture of Experts

Efektywne skalowanie modeli przez specjalizację expertów, a nie tylko parametry.

Czym jest MoE?

Mixture of Experts to architektura, w której każda warstwa transformera to nie jeden model, lecz zestaw specjalistów (expertów). Na każdym kroku router wybiera tylko kilku ekspertów do przetwarzania danego tokenu.

Kluczowa idea: model ma miliony expertów, ale dla jednego tokena używa ich tylko kilku. To pozwala mieć dużo większy model przy tym samym koszcie inferencji. 

# Zamiast:
# FFN(x) = W2 · ReLU(W1 · x)

# MoE:
# FFN_MoE(x) = Σ_i (g(x)_i · Expert_i(x))
# gdzie g(x) to router — wybiera top-k ekspertów dla każdego tokenu

Jak to działa?

  1. Router (gating network) — mała sieć neuronowa, która decyduje które experty wybrać dla danego tokenu
  2. Top-K selection — wybieramy zwykle K=1 lub K=2 ekspertów z N dostępnych
  3. Symetryczna dystrybucja obciążenia — balanced dispatch, żeby żadien ekspert nie był przeciążony
class MoELayer(nn.Module):
    def __init__(self, num_experts=8, top_k=2, d_model=768):
        super().__init__()
        self.router = nn.Linear(d_model, num_experts)  # gating
        self.experts = nn.ModuleList([nn.Sequential(
            nn.Linear(d_model, d_model),
            nn.GELU(),
            nn.Linear(d_model, d_model)
        ) for _ in range(num_experts)])
        self.top_k = top_k

    def forward(self, x):  # (batch_size, seq_len, d_model)
        gates = torch.softmax(self.router(x), dim=-1)  # wywagi dla ekspertów
        top_k_gates, top_k_indices = torch.topk(gates, self.top_k, dim=-1)

        output = torch.zeros_like(x)
        batch_size, seq_len, _ = x.shape

        for i in range(self.top_k):  # dla każdego z top-k ekspertów
            mask = (top_k_indices == i)  # które tokeny trafiają do eksperta
            expert_input = x[mask]  # tylko te tokeny
            if expert_input.numel() > 0:
                expert_output = self.experts[i](expert_input)
                weight = top_k_gates[mask, i].unsqueeze(-1)
                output[mask] += weight * expert_output

        return output

Dlaczego MoE?

  • Model 10x większy przy tym samym koszcie inferencji — GShard (Google) pokazał, że można skalować do setek miliardów parametrów przy jednym tokonie GPU compute
  • Specjalizacja eksperty — każdy ekspert może się "nauczyć" innego rodzaju tokena czy zadania
  • Efektywność przy długich kontekstach — większy model nie oznacza wolniejszej inferencji (jeśli K jest małe)

Przykłady modeli MoE

  • Mixtral 8x7B — 8 expertów na warstwę, tylko 2 aktywne przy inferencji (~45B params total, ~13B aktywnych)
  • GShard / Switch Transformer — wczesne architektur MoE z Google/DeepMind

Podsumowanie

MoE to kluczowa technika do budowania bardzo dużych modeli bez wybuchowego wzrostu kosztów inferencji. Model ma setki miliardów parametrów, ale dla każdego tokona obliczają się tylko kilkanaście miliardów — reszta ekspertów jest "wyłączona". Mixtral 8x7B to najlepszy przykład: model o pojemności ~45B parametrów przy kosztach inferencji ~13B.

Plusy

  • Duży model przy niskim koszcie inferencji
  • Szybka konwersacja (mniej tokenów do przetwarzania)
  • Naturalna specjalizacja — eksperty uczą się różnych rzeczy

Minusy

  • Duża złożoność implementacyjna (dispatch, balanced routing)
  • Ryzyko "expert collapse" — niektórzy eksperci nigdy się nie uczą
  • Precyzja może być niższa niż pełny dense model o podobnym rozmiarze
