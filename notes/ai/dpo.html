DPO — Direct Preference Optimization

Nowoczesna alternatywa dla RLHF — bez reward modelu i PPO.

Czym jest DPO?

DPO to metoda dopasowania preferencji, która pomija potrzebę reward modelu i PPO. Zamiast dwóch faz (trening reward modelem + RL) dostajesz jedną fazę — bezpośrednią optymalizację modelu na parach odpowiedzi "wygrana vs przegrana".

Kluczowy wniosek z artykułu DPO: można pokazać, że problem dopasowania preferencji ma rozwiązanie w zamkniętej formie — wystarczy zmodyfikować funkcję straty SFT i dodać do niej regularyzację KL względem modelu bazowego.

Jak to działa?

DPO trenuje na parach (wygrana odpowiedź, przegrana odpowiedź) dla każdego promptu. Funkcja straty porównuje prawdopodobieństwa wygenerowania obu odpowiedzi przez model i nakazuje, by wygrana miała większe prawdopodobieństwo — z regularyzacją KL względem modelu bazowego. 

# W skrócie: L_DPO = -log σ(β · log(p_winned/p_ref) - β · log(p_loser/p_ref))
# β — temperatura (kontrola odchylenia od modelu bazowego)

Przykład — DPO z TRL

Potrzebujesz tylko datasetu w formacie "wygrana vs przegrana" i SFT model:

from trl import DPOTrainer
from transformers import TrainingArguments

trainer = DPOTrainer(
    model=model,               # model po SFT (lub wstępnie wytrenowany)
    ref_model=None,            # model referencyjny — automatycznie z modelu
    train_dataset=dataset,     # format: {"prompt": ..., "chosen": ..., "rejected": ...}
    tokenizer=tokenizer,
    args=TrainingArguments(
        output_dir="./dpo-output",
        per_device_train_batch_size=4,
        num_train_epochs=3,
        learning_rate=5e-6,    # DPO potrzebuje małego lr
        fp16=True,
    ),
    beta=0.1,                   # temperatura — kontroluje odchylenie od modelu bazowego
)

Format danych do DPO

{  "prompt": "Jak napisać funkcję sortującą?",
  "chosen": "# Użycie wbudowanej metody\nlista.sort()\n# lub\ndef sortuj(tablica):\n    return sorted(tablica)",
  "rejected": # Kod z błędem lub słabą praktyką (np. implementacja bubble sort, brak komentarzy)
}

DPO vs RLHF

AspektRLHF (PPO)DPO
Ilość modeli3 (SFT + RM + PPO)1 (SFT → DPO)
ZłożonośćWysoka — stabilność PPONiska — jeden pipeline
Koszt danychOceny par (ludzie)Oceny par (ludzie — to samo!)
JakośćDobra, ale niestabilnaCzęsto lepsza i stabilniejsza

Kiedy DPO?

  • Dobierasz styl odpowiedzi firmy — to samo co RLHF, ale prościej wdrożyć
  • Nie masz zasobów na trening reward modelem i PPO (pamięć GPU)
  • Chcesz zbudować pipeline od SFT do alignmentu bez zbędnych komponentów

Podsumowanie

DPO to prostsza i często lepsza alternatywa dla RLHF. Zamiast trzech modeli i niestabilnego PPO, dostajesz jedną fazę optymalizacji na parach odpowiedzi. Wymaga tego samego zbioru danych co RLHF (oceny ludzkie par odpowiedzi), ale daje stabilniejsze i szybsze rezultaty.

Plusy

  • Jeden model zamiast trzech — prostszy pipeline
  • Brak PPO = brak niestabilności treningu
  • Szybsze wdrożenie i mniej GPU hours
  • Często lepsza jakość niż RLHF przy tych samych danych

Minusy

  • Dane: nadal potrzebujesz ocen ludzkich (to samo co w RLHF)
  • Wrażliwość na hiperparametr β — źle dobrany = słabe wyniki
  • Mniej elastyczny niż PPO przy niestandardowych reward functionach
