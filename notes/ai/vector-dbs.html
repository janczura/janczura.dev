Vector Databases & Semantic Search

Szukanie po znaczeniu, nie po słowach kluczowych.

Czym wektory i semantyczny search?

Tradycyjne wyszukiwanie (BM25, ElasticSearch) szuka słów kluczowych. Semantic search szuka po znaczeniu: pytanie "jak napisać sortowanie w Pythonie" znajdzie dokument o kodzie `sorted()` nawet bez słów "sort", "Python". 

# Workflow:
# 1. Podziel dokumenty na chunki (np. po ~500 tokenów)
# 2. Zamień każdy chunk na embedding (vector) przez model
# 3. Zapisz wektory w bazie danych
# 4. Przy zapytaniu: wygeneruj embedding zapytania i znajdź najbliższe

Embeddings — jak to działa?

Model embeddingowy (np. text-embedding-3-small, sentence-transformers) zamienia tekst w wektor liczbowy ~1500 wymiarów. Teksty o podobnym znaczeniu mają podobne wektory — bliskie kąta kosinusowego lub odległości euklidesowej. 

import openai, numpy as np

client = openai.OpenAI()

# Generowanie embeddingu
response = client.embeddings.create(
    model="text-embedding-3-small",
    input="Jak napisać sortowanie w Pythonie?"
)
query_embedding = response.data[0.embedding]

# Wyszukiwanie (znajdź wektory najbliższe)
cosine_similarities = [
    np.dot(query, doc_vec) / (np.linalg.norm(query) * np.linalg.norm(doc_vec))
    for doc_vec in document_vectors
]
top_results = np.argsort(cosine_similarities)[::-1][:5]  # top-5 wyników

Bazy wektorowe

  • Pinecone — managed, skalowalna, płatna (za dużo danych)
  • ChromaDB — open source, local-first, idealna do prototypów i małych projektów
  • Milvus / Qdrant — open source, produkcyjna skalowalność
  • pgvector (PostgreSQL) — rozszerzenie PG do wektorów. Najprostsze wdrożenie jeśli masz już bazę.
# ChromaDB — minimalny przykład
import chromadb, openai
chroma_client = chromadb.Client()
collection = chroma_client.create_collection("my-docs")

docs = ["Pierwsza dokumentacja...", "Druga sekcja..."]
embeddings = [openai.embeddings.create(input=d).data[0.embedding] for d in docs]

collection.add(documents=docs, embeddings=embeddings)
results = collection.query(query_texts=["Jak napisać sortowanie?"], n_results=3)

Hybrid Search (BM25 + Vector)

Najlepsze rezultaty daje połączenie obu metod: BM25 dla precyzyjnych nazw/funkcji, vector search dla kontekstu.

Podsumowanie

Vector search to podstawa nowoczesnych systemów RAG. Embeddingi zamieniają tekst w wektory, które można porównywać. Hybrid search (BM25 + vector) daje najlepsze rezultaty. ChromaDB i pgvector są najprostszymi punktami startu.

Plusy

  • Szuka po znaczeniu, nie słowach — znajduje odpowiedzi na pytania bez dokładnie tych słów
  • Połączenie z RAG daje "wiedzę" modelowi spoza jego treningu
  • ChromaDB/pgvector — zero konfiguracji, działa lokalnie

Minusy

  • Jakość zależy od jakości chunkingu i embeddingów
  • Bazy wektorowe są mniej precyzyjne niż SQL dla konkretnej informacji
  • Koszt embeddingów API rośnie z liczbą dokumentów
