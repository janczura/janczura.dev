LoRA — Low-Rank Adaptation

Efektywny fine-tuning bez modyfikowania wag modelu bazowego.

Czym jest LoRA?

LoRA to technika, która zamiast aktualizować wszystkie wagі dużego modelu, dodaje małe adaptery niskiego rzędu (low-rank matrices) równolegle do istniejących warstw. Model bazowy pozostaje zamrożony — uczysz tylko te adaptery.

Matematycznie: zamiast uczyć macierz wag W o wymiarze d × k, rozkładasz aktualizację na dwie małe macierze A (d × r) i B (r × k), gdzie r ≪ min(d, k). Typowo r = 8, 16, 32. 

# Zamiast:
# h = W · x          → uczenie całego W (miliony parametrów)

# LoRA:
# h = W · x + (A · B) · x   → W zamrożone, uczysz tylko A i B

Dlaczego to działa?

Badania pokazują, że podczas fine-tuningu macierz aktualizacji wag ma niski "intrinsic rank" — czyli większość informacji zmieści się w małej liczbie wymiarów. LoRA to wykorzystuje: zamiast uczyć 7 miliardów parametrów, uczysz kilka milionów adapterów.

Przykład — LoRA z PEFT + TRL

Kluczowa różnica względem SFT: dodajesz LoraConfig i get_peft_model():

from peft import LoraConfig, get_peft_model, TaskType
from trl import SFTTrainer
from transformers import AutoModelForCausalLM, AutoTokenizer

model_name = "codellama/CodeLlama-7b-hf"
tokenizer = AutoTokenizer.from_pretrained(model_name)
model = AutoModelForCausalLM.from_pretrained(
    model_name,
    torch_dtype=torch.float16,
    device_map="auto"
)

# Konfiguracja LoRA
lora_config = LoraConfig(
    r=16,                          # rang adaptera
    lora_alpha=32,                 # skala mnożenia
    target_modules=["q_proj", "v_proj"],  # warstwy do adaptacji
    lora_dropout=0.05,
    bias="none",
    task_type=TaskType.CAUSAL_LM,
)

# Ovoj model w PEFT — teraz tylko adaptery są trenowane
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# → trainable params: 0.17 % (4.2M vs 7B)

dataset = load_dataset("json", data_files="./code_examples.jsonl")["train"]

trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    train_dataset=dataset,
    dataset_text_field="text",
    max_seq_length=2048,
    args=TrainingArguments(
        output_dir="./lora-output",
        num_train_epochs=3,
        per_device_train_batch_size=4,
        learning_rate=2e-4,
        fp16=True,
    ),
)

trainer.train()

# Zapisz tylko adaptery (kilka MB, nie GB)
model.save_pretrained("./lora-adapter")

Inferencja z adapterem LoRA

from peft import PeftModel

base_model = AutoModelForCausalLM.from_pretrained(
    "codellama/CodeLlama-7b-hf",
    torch_dtype=torch.float16,
)

# Wczytaj adapter na wierzch modelu bazowego
model = PeftModel.from_pretrained(base_model, "./lora-adapter")
model.eval()

inputs = tokenizer("Stwórz funkcję sortującą:", return_tensors="pt")
output = model.generate(**inputs, max_new_tokens=256)
print(tokenizer.decode(output[0]))

Scenariusz: styl kodu repozytorium

LoRA jest najlepszym wyborem do tego zadania. Dlaczego:

  • Trening na jednym GPU (nawet 8GB VRAM z quantizacją)
  • Szybki — godziny zamiast dni
  • Adapter waży kilka MB — łatwy do wymiany i testowania
  • Możesz mieć osobny adapter na każdy projekt / styl kodu

Kluczowe hyperparametry LoRA

  • r (rang) — 8, 16, 32. Większe = więcej wyrazistości, ale wolniej i więcej pamięci
  • lora_alpha — skala; typowo 2 × r. Nie wpływa na liczbę parametrów
  • target_modules — które warstwy adaptować. q_proj, v_proj to standard; można dodać k_proj, o_proj, gate_proj, up_proj, down_proj
  • lora_dropout — regularizacja, typowo 0.05–0.1

Podsumowanie

LoRA to de facto standard fine-tuningu dużych modeli. Mrozi model bazowy, uczy tylko małe adaptery niskiego rzędu — oszczędność pamięci do 100× i czasu treningu do 10× względem pełnego SFT. Adaptery są lekkie (kilka MB), łatwe do wymiany i można je stosować na dowolnym modelu bazowym.

Plusy

  • Oszczędność pamięci — trening na jednym GPU 8GB+
  • Szybki trening (godziny vs dni)
  • Adaptery waży kilka MB, nie GB
  • Brak catastrophic forgetting — model bazowy jest nienaruszony
  • Możliwość wielu adapterów na jeden model

Minusy

  • Dodatkowy overhead inferencji (nawet mały)
  • Jakość może być nieco gorsza niż pełny SFT przy bardzo specyficznych zadaniach
  • Trzeba dobrać target_modules pod architekturę modelu
  • Adapter jest związany z modelem bazowym — nie przenosisz go na inny model
